# Python Baseline

适用范围：所有 Python 模块、CLI、services、runtime、MCP 相关代码。

## 目标

1. 保持可读性优先
2. 保持 agent 可理解性优先
3. 保持边界清晰、职责稳定

## 基线规则

1. Python 版本基线：3.10+
2. 新逻辑优先写在 `agents_memory/services/`，而不是 `scripts/` 包装层
3. 命令分发逻辑只放在 `agents_memory/commands/`
4. 路径解析、logger、bootstrap 只放在 `agents_memory/runtime.py`
5. 不允许把复杂业务逻辑重新塞回 `scripts/*.py`
6. 模块职责必须单一，优先小文件而不是大杂烩

## 可维护性要求

1. 优先显式数据流，避免隐式全局状态
2. 优先命名清晰的 helper，而不是堆叠长函数
3. I/O 与纯逻辑尽量分离，便于单元测试
4. 公共行为抽到共享函数，不复制粘贴逻辑
5. 复杂的代码逻辑必须写清解释性注释，帮助人和 agent 快速理解为什么这样分支、状态如何流动、哪一段是高风险决策

## 复杂度与重构约束

1. 当代码复杂度过高时，不允许继续在原函数或原模块上叠加分支，必须先重构再继续加功能
2. 复杂度判断优先服务于可读性、可验证性和 agent 可理解性，而不是只看单一数字
3. 命中任一硬性条件，直接判定为高复杂度，必须重构
4. 未命中硬性条件但命中三条及以上软性条件，也判定为高复杂度，应优先重构

## 高复杂度硬性条件

1. 单个函数或方法有效逻辑超过 40 行
2. 嵌套深度达到 4 层或以上
3. 单个函数同时混合 orchestration、domain decision、I/O 三类职责
4. 同一段逻辑需要跨越两个以上数据模型或上下文来回切换才能理解

## 高复杂度软性条件

1. 分支点超过 5 个，例如 `if / elif / match / except / early return`
2. 局部变量超过 8 个，且含义不再一眼可见
3. 同一个函数里出现两次以上“先查状态、再改状态、再写文件/回写结果”的重复节奏
4. 需要滚屏才能看完主路径，且主路径没有被 helper 折叠
5. 阅读者无法用一句话说清这个函数“只负责什么”
6. 新需求只能通过继续复制已有分支来接入，而不是复用共享 helper

## 重构动作模板

当复杂度被判定为高时，至少执行下面模板中的两项：

1. 拆出纯逻辑 helper，让分支判断和副作用分离
2. 拆出数据结构或 result object，减少平铺参数和临时变量
3. 把重复的 command / state / file update 流程抽到共享函数
4. 用显式阶段函数表达主流程，例如 `collect -> decide -> execute -> verify -> persist`
5. 为拆分后的关键路径补最小回归测试
6. 如果复杂逻辑暂时不能继续拆分，至少要在对应代码块前补一段简短注释，说明决策意图、状态约束或危险边界
