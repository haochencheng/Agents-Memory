## 错误上下文

**任务目标：**
修复 Vite/esbuild 的语法解析错误（`Unexpected ")"` at line 265），文件为
`packages/web/src/hooks/use-control-plane-projects-page.ts`。

**出错文件 / 位置：**
`Spec2Flow/packages/web/src/hooks/use-control-plane-projects-page.ts`
- 语法错误：line 265（重复的函数返回类型签名）
- 类型错误：line 477（`taskId: approval.taskId as string`）
- 类型错误：line 509（自动审批路径直接访问 `autoApprovalTarget.taskId`）

## 错误描述

1. `buildExecutionFeed(...)` 函数存在**重复的返回类型签名行**（merge 残留），导致 esbuild 解析失败：
   ```
   [plugin:vite:esbuild] Transform failed with 1 error: Unexpected ")"
   ```

2. 删除重复行后，TypeScript 报 2 处类型错误：
   ```
   Type 'string | null | undefined' is not assignable to type 'string'
   ```
   原因：`approval.taskId` 在 API 类型定义中是 `string | null | undefined`，
   但下游的 `PendingConfirmationItem.taskId` 和 `taskActionMutation.mutate` 的 payload 都要求 `string`。

## 根因分析

1. 语法错误：代码合并时保留了两行函数签名，esbuild 不容忍这种重复。
2. 类型错误的**真正根因**：
   - 用了运行时 `.filter()` 做类型收窄（`typeof approval.taskId === 'string'`），
     但 TypeScript 不认 `.filter()` 作为类型守卫，数组元素类型依然保持宽泛。
   - 后续用 `as string` 断言来哑火错误，而不是在源头收窄类型。
   - 删掉断言后，链路上所有消费点同时暴露这个本来就存在的类型漏洞。

## 修复方案

1. 删除重复的函数返回类型行（保留使用 `FeedTone` 别名的那一行）。
2. 新增专用类型守卫函数，替代 `.filter()` 的运行时检查：
   ```ts
   function isRequestedApprovalWithTaskId(
     approval: PlatformObservabilityApprovalItem,
   ): approval is PlatformObservabilityApprovalItem & { status: 'requested'; taskId: string } {
     return approval.status === 'requested'
       && typeof approval.taskId === 'string'
       && approval.taskId.length > 0;
   }
   ```
3. 在 `.filter()` 中使用类型守卫，后续 `.map()` 里的 `taskId` 自动推导为 `string`。

## 提炼规则

**`.filter()` 不是类型守卫**：若需要在 TypeScript 中通过过滤收窄数组元素类型，
必须使用 `(x): x is T` 形式的谓词函数，不能依赖 `.filter()` 的运行时逻辑做类型收窄。
用 `as T` 断言来哑火类型错误是技术债，根因未解决。
